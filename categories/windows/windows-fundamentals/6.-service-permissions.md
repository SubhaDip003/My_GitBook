# 6. Service Permissions

**Service Permissions** refer to the access rights and configurations that control how Windows services run and who/what they run as. Services are background processes (like DHCP, AD DS, etc.) that keep the system and network functioning. These services run using a specific user account, and that account’s permissions determine what the service can do.

A common issue happens when services are configured to run using a normal user account (for example, “Bob”). If that account is later disabled or removed (like when the employee leaves), the service will fail to start. This can cause serious problems, such as DHCP failing and devices not getting IP addresses, leading to network downtime. That’s why important services should always run using dedicated **service accounts**, not personal user accounts.

From a security perspective, service permissions are very important because misconfigurations can be exploited. If a service runs with high privileges and its executable path or folder has weak permissions, an attacker can replace the original executable or DLL with a malicious one. When the service starts, it will execute the attacker’s code with elevated privileges. This can lead to **privilege escalation, persistence, or remote code execution**.

***

## Examining Services using services.msc <a href="#examining-services-using-servicesmsc" id="examining-services-using-servicesmsc"></a>

**Examining Services using `services.msc`** means viewing and managing Windows services through a graphical interface. When you open `services.msc`, you can see important details about each service such as the **service name, display name, startup type, status, and path to the executable**. The service name is especially useful because it is used in command-line tools to manage services. The **path to executable** shows the exact file that runs when the service starts. If this file or its folder has weak NTFS permissions, an attacker could replace it with a malicious file and gain execution.

<figure><img src="../../../.gitbook/assets/image (1) (1).png" alt=""><figcaption></figcaption></figure>

Most Windows services run using special built-in accounts like&#x20;

* **LocalSystem**
* **LocalService,**
* **NetworkService**.&#x20;

Among these, **LocalSystem has the highest privileges**, meaning it has almost complete control over the system. Because of this, if a service running as LocalSystem is compromised, it can lead to full system compromise. That is why services should follow the **principle of least privilege**, meaning they should run with only the permissions they actually need.

[Here is one breakdown of the principle of least privilege](https://www.cloudflare.com/learning/access-management/principle-of-least-privilege/)

The **Recovery tab** in a service is also important. It defines what happens if the service fails, such as restarting the service or running a program. If misconfigured, this can be abused by an attacker to execute malicious programs automatically when the service fails. Overall, analyzing services using `services.msc` helps identify weak configurations, privilege risks, and possible attack paths in a Windows system.

## Examining services using sc <a href="#examining-services-using-sc" id="examining-services-using-sc"></a>

The **`sc` (Service Control)** command is a Windows command-line tool used to view, manage, and modify services. It is very useful for both system administration and security analysis because it allows you to interact with services without using a graphical interface.

```cmd
sc qc wuauserv

[SC] QueryServiceConfig SUCCESS

SERVICE_NAME: wuauserv
TYPE               : 20  WIN32_SHARE_PROCESS
START_TYPE         : 3   DEMAND_START
ERROR_CONTROL      : 1   NORMAL
BINARY_PATH_NAME   : C:\WINDOWS\system32\svchost.exe -k netsvcs -p
DISPLAY_NAME       : Windows Update
DEPENDENCIES       : rpcss
SERVICE_START_NAME : LocalSystem
```

This command shows the **configuration of the service**. The most important part here is the **BINARY\_PATH\_NAME**, which tells which executable runs the service. If this path is writable or changeable, it can be abused by attackers. Also notice **SERVICE\_START\_NAME = LocalSystem**, which means the service runs with the highest privilege.

#### Start and Stop Services

We can also use sc to start and stop services.

The command **`sc stop wuauserv`** is used to stop a running service. However, stopping services usually requires administrative privileges. If the command returns “Access is denied,” it means the current user does not have sufficient permissions.

#### Change the service executable

The command **`sc config wuauserv binPath=...`** is used to change the executable path of a service. This is very important from a security perspective because if an attacker has permission to modify the service configuration, they can replace the original executable with a malicious program. When the service starts, the malicious program will run with the service’s privileges, often as SYSTEM.

If someone has permission, they can change the service executable:

```cmd
C:\WINDOWS\system32> sc config wuauserv binPath=C:\Winbows\Perfectlylegitprogram.exe
```

The command **`sc qc wuauserv`** is used to check the configuration of a service. It shows details like the service name, startup type, executable path, and the account under which the service runs (for example, LocalSystem). This information is important because the executable path tells you which program is running as a service, and if that path is writable or misconfigured, it can be abused.

The command **`sc \\hostname query ServiceName`** is used to query a service on a remote system over the network. This allows administrators or attackers to check services running on another machine, which is useful in both management and lateral movement scenarios.

#### View the security permissions of a service

The command **`sc sdshow wuauserv`** is used to view the security permissions of a service in a format called SDDL (Security Descriptor Definition Language). This output defines which users or groups can perform actions on the service. It includes permissions like starting the service, stopping it, or reading its configuration. In this format, values like “AU” represent Authenticated Users, and “A” means access is allowed.

In Windows, every service is a securable object and has a security descriptor. This descriptor contains a **DACL (Discretionary Access Control List)**, which defines who can access the service and what actions they can perform, and a **SACL (System Access Control List)**, which is used for logging and auditing access attempts. If the DACL is weak or misconfigured, it may allow low-privileged users to control or modify services.

```cmd
C:\WINDOWS\system32> sc sdshow wuauserv

D:(A;;CCLCSWRPLORC;;;AU)
 (A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;BA)
 (A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;SY)
S:(AU;FA;CCDCLCSWRPWPDTLOSDRCWDWO;;;WD)
```

This looks complex, but it is just **permissions of the service written in a special format called SDDL**.

Focus on this part:

```
D:(A;;CCLCSWRPLORC;;;AU)
```

This means:

* **D:** → This is the **DACL (permissions)**
* **(A;;...;;;AU)** → One permission entry

```
D:  → The following part contains DACL (permission rules)

AU: → Authenticated Users (all logged-in users)

A;; → Access is Allowed

CC  → SERVICE_QUERY_CONFIG (can check service configuration)

LC  → SERVICE_QUERY_STATUS (can check service status)

SW  → SERVICE_ENUMERATE_DEPENDENTS (can list dependent services)

RP  → SERVICE_START (can start the service)

LO  → SERVICE_INTERROGATE (can query current state)

RC  → READ_CONTROL (can read permissions/security settings)
```

**Authenticated Users are allowed to read and query the service and even start it.**

## Simple Understanding of SDDL

**Security Descriptor Definition Language (SDDL)** is a text format used in Windows to represent **permissions of an object** (like files, folders, services, registry keys). Instead of showing permissions in GUI form, Windows converts them into a **compact string** that defines who can access the object and what actions they can perform.

Every securable object in Windows has a **Security Descriptor**, and SDDL is just a readable way to display it. A security descriptor mainly contains:

* **Owner (O:)**
* **Group (G:)**
* **DACL (D:) → Who is allowed/denied access**
* **SACL (S:) → Logging/auditing rules**

#### 🔹 Example of SDDL

```
D:(A;;CCLCSWRPLORC;;;AU)
```

Now understand this step by step in simple way:

```
D:   → This part defines DACL (access permissions)

(A;;CCLCSWRPLORC;;;AU)
```

Breakdown of this block:

* **A** → Access is Allowed
* **;;** → Format separator
* **CCLCSWRPLORC** → Permissions (actions allowed)
* **;;;AU** → Applies to Authenticated Users

#### 🔹 Meaning of Permissions

```
CC → Query service config  
LC → Query service status  
SW → Enumerate dependent services  
RP → Start service  
LO → Interrogate service  
RC → Read permissions
```

#### 🔹 Simple Meaning of Full Example

```
D:(A;;CCLCSWRPLORC;;;AU)
```

👉 Authenticated Users are **allowed** to:

* Check service configuration
* Check status
* Start the service
* Read its permissions

#### 🔹 Another Example (Multiple Entries)

```
D:(A;;CCLCSWRPLORC;;;AU)(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;BA)
```

This means:

* First part → Permissions for **Authenticated Users (AU)**
* Second part → Permissions for **Administrators (BA)**

Each bracket `()` is called an **Access Control Entry (ACE)**.

## Examine service permissions using PowerShell <a href="#examine-service-permissions-using-powershell" id="examine-service-permissions-using-powershell"></a>

Using the `Get-Acl` PowerShell cmdlet, we can examine service permissions by targeting the path of a specific service in the registry.

```powershell
PS C:\Users\htb-student> Get-ACL -Path HKLM:\System\CurrentControlSet\Services\wuauserv | Format-List

Path   : Microsoft.PowerShell.Core\Registry::HKEY_LOCAL_MACHINE\System\CurrentControlSet\Services\wuauserv
Owner  : NT AUTHORITY\SYSTEM
Group  : NT AUTHORITY\SYSTEM
Access : BUILTIN\Users Allow  ReadKey
         BUILTIN\Users Allow  -2147483648
         BUILTIN\Administrators Allow  FullControl
         BUILTIN\Administrators Allow  268435456
         NT AUTHORITY\SYSTEM Allow  FullControl
         NT AUTHORITY\SYSTEM Allow  268435456
         CREATOR OWNER Allow  268435456
         APPLICATION PACKAGE AUTHORITY\ALL APPLICATION PACKAGES Allow  ReadKey
         APPLICATION PACKAGE AUTHORITY\ALL APPLICATION PACKAGES Allow  -2147483648
         S-1-15-3-1024-1065365936-1281604716-3511738428-1654721687-432734479-3232135806-4053264122-3456934681 Allow
         ReadKey
         S-1-15-3-1024-1065365936-1281604716-3511738428-1654721687-432734479-3232135806-4053264122-3456934681 Allow
         -2147483648
Audit  :
Sddl   : O:SYG:SYD:AI(A;ID;KR;;;BU)(A;CIIOID;GR;;;BU)(A;ID;KA;;;BA)(A;CIIOID;GA;;;BA)(A;ID;KA;;;SY)(A;CIIOID;GA;;;SY)(A
         ;CIIOID;GA;;;CO)(A;ID;KR;;;AC)(A;CIIOID;GR;;;AC)(A;ID;KR;;;S-1-15-3-1024-1065365936-1281604716-3511738428-1654
         721687-432734479-3232135806-4053264122-3456934681)(A;CIIOID;GR;;;S-1-15-3-1024-1065365936-1281604716-351173842
         8-1654721687-432734479-3232135806-4053264122-3456934681)

```

This command is showing the **permissions of a Windows Registry key** (`wuauserv` service) in a readable format along with its raw SDDL string. The important parts are **Owner, Access, and Sddl**.

The **Owner and Group** are both `NT AUTHORITY\SYSTEM`, which means the SYSTEM account owns this registry key and has full control over it. This is normal for critical Windows services because SYSTEM has the highest privilege on the machine.

The **Access section** shows which users or groups can do what. For example, `BUILTIN\Users Allow ReadKey` means normal users can only read this registry key but cannot modify it. `BUILTIN\Administrators Allow FullControl` means administrators can fully control this key, including modifying or deleting it. `NT AUTHORITY\SYSTEM Allow FullControl` confirms SYSTEM also has full access. You also see `APPLICATION PACKAGE AUTHORITY\ALL APPLICATION PACKAGES`, which allows apps (like UWP apps) to read the key. Some entries show numbers like `268435456` or `-2147483648`; these are just numeric representations of specific permissions internally used by Windows and are equivalent to certain rights like read or generic access.

The **Sddl part** is the same permission information but written in a compact coded format. For example:

```
O:SYG:SY
```

means Owner = SYSTEM and Group = SYSTEM.

Then:

```
D:AI(...)
```

means this is the DACL (permissions), and it is inherited (AI = Auto Inherited).

Inside the brackets:

```
(A;ID;KR;;;BU)
```

means:

* **A** → Allow
* **ID** → Inherited permission
* **KR** → Key Read (read access)
* **BU** → Built-in Users

So this line means: Users are allowed to read the registry key.

Another example:

```
(A;ID;KA;;;BA)
```

means:

* **KA** → Key All Access (full control)
* **BA** → Built-in Administrators

So administrators have full control.

Similarly:

```
(A;ID;KA;;;SY)
```

means SYSTEM has full control.

Overall, this output shows that **normal users can only read the service registry key, while administrators and SYSTEM have full control**, which is a secure and expected configuration. The SDDL is just a compressed version of the same permissions, and once you understand the codes, you can quickly identify who has what level of access.

Notice how this command returns specific account permissions in an easy-to-read format and in SDDL. Also, the SID that represents each security principal (User and/or Group) is present in the SDDL. This is something we do not get when running `sc` from the command prompt.

Knowing how to interact with services and their associated permissions from the command line makes it easier to script out these tasks. While it is good to know how to perform these tasks from the GUI, it does not scale well as we start getting into larger network environments and Domains.
